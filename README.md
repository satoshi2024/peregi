### 2. QA29912 测试数据具体准备方案（对应你的式样书）
你需要找到系统底层的课税信息表（就是游标把数据塞进 RDATA 的那个源头表）。
#### 🟢 **Case 006-001 (验证全额减免时，强制清零)**
你需要造一条**“虽然有税额，但是被减免完全抵消”**，并且**“原本会产生尾数 (端数)”**的数据。
 * **源表关键字段设定（市、县都要设）：**
   * 均等割额 (SHIM_KINTOWARI, KEN_KINTOWARI) = **5000**
   * 均等割减免额 (SHIM_GENMEN_KINWARI, KEN_GEN_KINTOWARI) = **5000**
   * 所得割额 (SHIM_SHOTOKUWARI, KEN_SHOTOKUWARI) = 10000 （随便给个正数）
   * 所得割减免额 (SHIM_GENMEN_SHOWARI, KEN_GEN_SHOWARI) = 10000 （全部减免掉）
   * **🎯 核心关键：尾数/端数 (SHIM_HASU, KEN_HASU) = 100 （必须给一个大于0的数！）**
> **💡 为什么要特别设定端数？**
> 因为原本的 Bug 就是“在全额减免时，多加了端数导致金额不为 0”。如果你造的数据本来就没有端数（端数=0），那你不用修代码跑出来也是 0，测试就没有说服力了。设定端数为 100，如果在你的新代码下跑出来依然是 0，这就**铁证如山**地证明了你的 IF 拦截逻辑生效了，把那个 100 给强行抹掉了。
> 
#### 🟡 **Case 006-002 (验证不符合全额减免时，保留原有逻辑)**
你需要造一条普通的、需要交纳均等割的数据，证明你新加的 IF 没有误伤正常人。
 * **源表关键字段设定：**
   * 均等割额 (SHIM_KINTOWARI, KEN_KINTOWARI) = **5000**
   * 均等割减免额 (SHIM_GENMEN_KINWARI, KEN_GEN_KINTOWARI) = **0** *(或者 1000，只要差值不是 0 就行)*
   * 所得割额 (SHIM_SHOTOKUWARI, KEN_SHOTOKUWARI) = 10000
   * 所得割减免额 (SHIM_GENMEN_SHOWARI, KEN_GEN_SHOWARI) = 2000
   * 尾数/端数 (SHIM_HASU, KEN_HASU) = **100**
> **💡 预期结果：**
> 跑完程序后，你去查中间表，这个用户的所得割额应该会等于 (10000 - 2000) + 100 = 8100。这就证明了 ELSE 分支正常运行，原有的端数计算逻辑没有被破坏。
>
